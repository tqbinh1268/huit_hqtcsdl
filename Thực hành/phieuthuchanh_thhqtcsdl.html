PHẦN I: THIẾT LẬP CƠ SỞ DỮ LIỆU KIỂM THỬ (QL_SINHVIEN)

Sinh viên thực hiện sao chép toàn bộ đoạn script lệnh T-SQL dưới đây và chạy trên phần mềm SQL Server Management Studio (SSMS). Hệ thống dữ liệu này được xây dựng có chủ đích nhằm tái hiện các trạng thái dữ liệu đa chiều phục vụ cho các thuật toán gộp nhóm nâng cao, bẫy lỗi trigger, con trỏ cursor dữ liệu hàng dọc và quản lý tranh chấp đồng thời. 

-- =========================================================================
-- 1. TẠO CƠ SỞ DỮ LIỆU (DATABASE) VỚI CẤU HÌNH DUNG LƯỢNG VÀ FILE HỆ THỐNG
-- =========================================================================
USE master;
GO

IF EXISTS (SELECT * FROM sys.databases WHERE name = 'QL_SINHVIEN')
BEGIN
    ALTER DATABASE QL_SINHVIEN SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
    DROP DATABASE QL_SINHVIEN;
END
GO

CREATE DATABASE QL_SINHVIEN
ON PRIMARY (
    NAME = QLSV_PRIMARY,
    FILENAME = 'C:\Program Files\Microsoft SQL Server\MSSQL16.MSSQLSERVER\MSSQL\DATA\QLSV_PRIMARY.mdf',
    SIZE = 20MB,
    MAXSIZE = 100MB,
    FILEGROWTH = 10%
)
LOG ON (
    NAME = QLSV_LOG,
    FILENAME = 'C:\Program Files\Microsoft SQL Server\MSSQL16.MSSQLSERVER\MSSQL\DATA\QLSV_LOG.ldf',
    SIZE = 10MB,
    MAXSIZE = 50MB,
    FILEGROWTH = 15%
);
GO

USE QL_SINHVIEN;
GO

-- =========================================================================
-- 2. TẠO CẤU TRÚC CÁC BẢNG (TABLES) VÀ CÁC RÀNG BUỘC TOÀN VẸN CHẶT CHẼ
-- =========================================================================

CREATE TABLE KHOA (
    MAKHOA VARCHAR(10) NOT NULL,
    TENKHOA NVARCHAR(50) NOT NULL,
    CONSTRAINT PK_KHOA PRIMARY KEY (MAKHOA)
);

CREATE TABLE LOP (
    MALOP VARCHAR(10) NOT NULL,
    TENLOP NVARCHAR(50) NOT NULL,
    SISO INT DEFAULT 0,
    MAKHOA VARCHAR(10),
    CONSTRAINT PK_LOP PRIMARY KEY (MALOP),
    CONSTRAINT FK_LOP_KHOA FOREIGN KEY (MAKHOA) REFERENCES KHOA(MAKHOA),
    CONSTRAINT CK_SISO CHECK (SISO >= 0)
);

CREATE TABLE SINHVIEN (
    MASV VARCHAR(10) NOT NULL,
    HOTEN NVARCHAR(50) NOT NULL,
    NGSINH DATE,
    DCHI NVARCHAR(100),
    GIOITINH NVARCHAR(3),
    MALOP VARCHAR(10),
    DIEMTB DECIMAL(3,1) DEFAULT NULL,
    XEPLOAI NVARCHAR(10) DEFAULT NULL,
    CONSTRAINT PK_SINHVIEN PRIMARY KEY (MASV),
    CONSTRAINT FK_SINHVIEN_LOP FOREIGN KEY (MALOP) REFERENCES LOP(MALOP),
    CONSTRAINT CK_GIOITINH CHECK (GIOITINH IN (N'Nam', N'Nữ'))
);

CREATE TABLE MONHOC (
    MAMH VARCHAR(10) NOT NULL,
    TENMH NVARCHAR(50) NOT NULL,
    SOTC INT NOT NULL,
    BATBUOC BIT DEFAULT 1,
    CONSTRAINT PK_MONHOC PRIMARY KEY (MAMH),
    CONSTRAINT CK_SOTC CHECK (SOTC > 0)
);

CREATE TABLE KETQUA (
    MASV VARCHAR(10) NOT NULL,
    MAMH VARCHAR(10) NOT NULL,
    HOCKY INT NOT NULL,
    DIEMTHI DECIMAL(3,1),
    CONSTRAINT PK_KETQUA PRIMARY KEY (MASV, MAMH, HOCKY),
    CONSTRAINT FK_KETQUA_SINHVIEN FOREIGN KEY (MASV) REFERENCES SINHVIEN(MASV) ON DELETE CASCADE,
    CONSTRAINT FK_KETQUA_MONHOC FOREIGN KEY (MAMH) REFERENCES MONHOC(MAMH),
    CONSTRAINT CK_DIEMTHI CHECK (DIEMTHI BETWEEN 0 AND 10)
);
GO

PHẦN II: HỆ THỐNG PHIẾU BÀI TẬP THỰC HÀNH HÀNG TUẦN

📑 PHIẾU BÀI TẬP SỐ 1: CẤU TRÚC VẬT LÝ, RÀNG BUỘC TOÀN VẸN VÀ VIEW NÂNG CAO

Mục 1: Khai thác cấu trúc vật lý hệ thống
  • Câu 1.1: Viết câu lệnh T-SQL kiểm tra danh sách tập tin vật lý (.mdf, .ldf) của CSDL QL_SINHVIEN đang đặt tại thư mục nào trên phân vùng đĩa. Xác định dung lượng hiện tại của Transaction Log dựa trên bảng hệ thống.
  • Câu 1.2: Thực hiện lệnh bổ sung một FILEGROUP mang tên DATA_SECONDARY vào CSDL. Sau đó gắn thêm một tập tin dữ liệu phụ QLSV_DATA2.ndf vào filegroup này với kích thước khởi tạo 5MB, tự tăng trưởng 10%.
  • Câu 1.3: Giả lập tình huống phân vùng lưu trữ nhật ký bị đầy. Hãy viết câu lệnh thu hẹp dung lượng tập tin log (QLSV_LOG) về cấu hình tối thiểu 2MB bằng lệnh DBCC SHRINKFILE.
Mục 2: Ràng buộc toàn vẹn nâng cao (Constraints)
  • Câu 1.4: Hãy bổ sung một ràng buộc kiểm tra (CHECK Constraint) trên bảng SINHVIEN sao cho dữ liệu ngày sinh (NGSINH) khi nhập vào phải đảm bảo sinh viên đó đủ từ 18 tuổi trở lên tính đến năm hiện hành của hệ thống máy chủ.
  • Câu 1.5: Thử nghiệm chèn một dòng sinh viên mới dính lỗi vi phạm ràng buộc khóa ngoại (Ví dụ chèn mã lớp '24DHTH99' chưa tồn tại trong bảng danh mục lớp). Hãy sao chép thông báo lỗi trả về của hệ thống và giải trình cơ chế chặn lỗi.
Mục 3: Bảng ảo (View) phối hợp biểu thức tính toán và hàm gộp nhóm
  • Câu 1.6: Khởi tạo bảng ảo tên là vw_DiemTrungBinhTichLuy hiển thị danh sách gồm: Mã sinh viên, Họ tên, Mã Lớp, Tổng số tín chỉ tích lũy (chỉ tính các môn đạt điểm thi $\ge 5.0$) và Điểm trung bình tích lũy hệ số 10 (Điểm TB = Tổng (Điểm thi $ imes$ Số tín chỉ) / Tổng số tín chỉ).
💡 Ngóc ngách logic cần giải quyết (Edge Case):
Sinh viên có quyền thi một môn học nhiều lần ở các học kỳ khác nhau để cải thiện điểm. Câu lệnh truy vấn trong View bắt buộc phải lọc lấy điểm thi lớn nhất (MAX) của môn học đó, tuyệt đối không được cộng gộp nhân đôi số tín chỉ môn học đó khi tính tổng tín chỉ tích lũy.
  • Câu 1.7: Thiết lập View vw_ThongKeHocVu liệt kê danh mục các lớp học kèm theo tỷ lệ sinh viên rớt môn. Chỉ hiển thị các lớp học có tỷ lệ sinh viên bị xếp loại rớt môn (sinh viên có tối thiểu một môn dưới 5.0 ở học kỳ gần nhất) chiếm tỷ lệ vượt quá 30% tổng sĩ số của lớp đó.

📑 PHIẾU BÀI TẬP SỐ 2: LẬP TRÌNH CƠ SỞ DỮ LIỆU (STORED PROCEDURE & FUNCTION)

Mục 1: Thủ tục lưu trữ (Stored Procedure)
  • Câu 2.1: Lập trình Stored Procedure tên là sp_ThemSinhVien nhận các tham số tương ứng với các cột của bảng SINHVIEN.
💡 Ngóc ngách nghiệp vụ bẫy lỗi đầu vào:
Trong thân thủ tục, trước khi chạy lệnh INSERT, sinh viên phải viết các khối kiểm tra:
1. Nếu MASV truyền vào đã tồn tại $ ightarrow$ Báo lỗi bằng RAISERROR và ngắt thủ tục bằng lệnh RETURN.
2. Nếu MALOP truyền vào chưa tồn tại trong bảng danh mục LOP $ ightarrow$ Báo lỗi lớp không hợp lệ và ngắt tiến trình.
3. Chỉ khi vượt qua các bước kiểm tra mới chèn bản ghi và in ra chuỗi "Thêm mới sinh viên thành công".
  • Câu 2.2: Xây dựng Stored Procedure tên là sp_LayThongKeLop tiếp nhận tham số đầu vào là @MaLop VARCHAR(10). Thủ tục phải kết xuất và trả về 2 giá trị dạng số nguyên ra màn hình thông qua tham số đầu ra (OUTPUT):
    1. Số lượng sinh viên thuộc giới tính Nam của lớp đó.
    2. Số lượng sinh viên thuộc giới tính Nữ của lớp đó.
    Viết đoạn lệnh khai báo các biến để gọi thực thi thủ tục này và xuất kết quả kiểm thử.
Mục 2: Hàm do người dùng định nghĩa (User-Defined Function)
  • Câu 2.3: Thiết lập hàm vô hướng (Scalar Function) mang tên fn_XepLoaiSinhVien nhận vào tham số điểm số @DiemTB DECIMAL(3,1). Hàm xử lý cấu trúc điều kiện trả về chuỗi ký tự NVARCHAR(10) quy định: Điểm $\ge 9.0$: "Xuất sắc", 8.0 - 8.9: "Giỏi", 6.5 - 7.9: "Khá", 5.0 - 6.4: "Trung bình", và dưới 5.0 xếp loại "Yếu".
  • Câu 2.4: Thiết lập hàm trả về cấu trúc bảng (Inline Table-Valued Function) có tên là fn_DanhSachHocLai. Hàm nhận vào hai tham số: @MaMonHoc VARCHAR(10) và @HocKy INT. Kết quả hàm trả về bảng chi tiết (Mã SV, Họ tên, Điểm thi) các sinh viên dính kết quả điểm dưới 5.0 của môn học và học kỳ chỉ định.

📑 PHIẾU BÀI TẬP SỐ 3: LẬP TRÌNH BẪY LỖI TỰ ĐỘNG VÀ CON TRỎ TRÊN BỘ DỮ LIỆU (TRIGGER & CURSOR)

Mục 1: Triggers quản lý tính toàn vẹn dữ liệu động
  • Câu 3.1: Cài đặt một AFTER INSERT Trigger đặt trên bảng dữ liệu SINHVIEN. Mỗi khi có bản ghi sinh viên mới được ghi nhận vào lớp, trigger phải tự động tính toán cộng thêm 1 vào cột sĩ số (SISO) của lớp học đó trong bảng LOP.
💡 Ngóc ngách kỹ thuật (Tránh bẫy dòng đơn):
Sinh viên tuyệt đối không dùng biến đơn để gán dữ liệu từ bảng ảo Inserted (như lệnh gán SELECT @MaLop = MALOP FROM Inserted), vì lệnh này sẽ chạy sai hoàn toàn khi giáo vụ thực hiện chèn đồng thời một danh sách gồm nhiều dòng sinh viên. Bản chất trigger phải được viết dựa trên câu lệnh toán tử gộp nhóm UPDATE LOP SET SISO = SISO + Count... FROM LOP JOIN Inserted... GROUP BY... để đảm bảo sĩ số luôn đồng bộ chính xác.
  • Câu 3.2: Cài đặt một AFTER DELETE Trigger cài đặt trên bảng SINHVIEN. Khi xóa bản ghi sinh viên, trigger tự động thực hiện trừ sĩ số lớp học đó đi 1 đơn vị. Đồng thời bổ sung logic: Nếu sĩ số lớp học sau khi trừ giảm về mức bằng 0, phát thông báo dòng chữ bằng lệnh PRINT nhằm cảnh báo lớp trống.
  • Câu 3.3: Cài đặt một AFTER UPDATE Trigger trên bảng KETQUA. Nếu cán bộ thực hiện chỉnh sửa trường điểm thi (DIEMTHI), trigger tiến hành kiểm tra xem giá trị điểm mới nạp có nằm trong khoảng thang chuẩn từ 0.0 đến 10.0 hay không. Nếu phát hiện vi phạm, thực hiện hủy bỏ giao tác quay lui bằng lệnh ROLLBACK TRANSACTION và báo lỗi.
Mục 2: Con trỏ duyệt dữ liệu hàng dọc (Cursor)
  • Câu 3.4: Viết một đoạn chương trình T-SQL hoàn chỉnh có sử dụng cấu trúc vòng lặp CURSOR duyệt qua danh mục tất cả các sinh viên trong bảng SINHVIEN. Tại mỗi dòng sinh viên duyệt qua, thực hiện chuỗi tác vụ:
    1. Tính toán điểm trung bình thi tất cả các môn của sinh viên đó từ bảng KETQUA.
    2. Gọi hàm vô hướng fn_XepLoaiSinhVien (ở Phiếu 2) để quy đổi ra chuỗi xếp loại tương ứng.
    3. Chạy lệnh cập nhật (UPDATE) trực tiếp giá trị tính được vào hai trường DIEMTB và XEPLOAI của chính dòng sinh viên con trỏ đang trỏ tới.
    Yêu cầu bắt buộc: Kiểm tra biến trạng thái @@FETCH_STATUS = 0 để điều khiển vòng lặp và gọi đầy đủ hai lệnh đóng con trỏ (CLOSE) và giải phóng bộ nhớ (DEALLOCATE) cuối lệnh.

📑 PHIẾU BÀI TẬP SỐ 4: QUẢN TRỊ BẢO MẬT VÀ PHÂN QUYỀN HỆ THỐNG (SECURITY)

Câu 4.1: Tạo lập tài khoản và cấp phát quyền cơ bản
Viết mã lệnh T-SQL thực hiện chuỗi hành động quản trị bảo mật an toàn thông tin sau:

Câu 4 (2.0 điểm) - Lập trình quản lý giao tác an toàn ACID lồng cấu trúc bẫy lỗi:
Viết đoạn mã lệnh T-SQL mô phỏng một giao tác chỉnh sửa dữ liệu an toàn. Tiến trình thực hiện đồng thời hai việc: Thêm mới một môn học tự chọn vào bảng MONHOC, sau đó nạp điểm thi cho một sinh viên ở bảng KETQUA. Toàn bộ chuỗi lệnh phải đặt trong khối điều khiển giao tác BEGIN TRANSACTION dính kèm khối bẫy lỗi hệ thống TRY...CATCH. Cấu hình sao cho nếu lệnh nạp điểm dính lỗi vi phạm khóa ngoại (Mã SV truyền vào không tồn tại), khối CATCH tự động gọi lệnh ROLLBACK TRANSACTION trả dữ liệu môn học tự chọn vừa thêm về trạng thái không có gì, giữ an toàn hệ thống.

Cài đặt Trigger kiểm soát dữ liệu liên bảng (30%) Trigger thiết kế chuẩn tối ưu, khai thác chính xác bảng ảo Inserted, hoạt động hoàn hảo khi chèn dữ liệu tệp nhiều dòng. Trigger vận hành đúng khi chèn đơn dòng, nhưng thuật toán dính lỗi hoặc báo lỗi hệ thống khi nhập liệu đồng thời nhiều dòng. Trigger định nghĩa đúng cấu trúc nhưng lọc sai mệnh đề JOIN bảng dẫn đến việc phát câu cảnh báo sai đối tượng môn học. Trigger dính lỗi hệ thống biên dịch cú pháp, không có khả năng tự động kích hoạt khi có sự kiện INSERT/UPDATE dữ liệu điểm. Không viết được mã lệnh hoặc viết sai lệch hoàn toàn bản chất nghiệp vụ của trigger kiểm soát.
Bảo mật hệ thống & Phân quyền tài khoản (20%) Tạo lập chính xác Login/User dữ liệu, cấp quyền GRANT đúng thực thể bảng và thiết lập chặn mức cột bằng lệnh DENY hoàn hảo. Tạo đúng Login/User và cấp quyền thao tác, nhưng cấu trúc câu lệnh chặn mức cột của lệnh DENY bị viết sai cú pháp. Nhầm lẫn bản chất giữa Login hệ thống máy chủ và User đối tượng database, gán quyền sai thực thể chỉ định. Chỉ khởi tạo được tài khoản rỗng, hoàn toàn bất lực không triển khai được các câu lệnh phân quyền cấp phát quản trị. Bỏ trống không làm nội dung bảo mật an toàn thông tin hoặc viết mã lỗi.
Chiến lược Sao lưu phòng ngừa & Di trú phục hồi (20%) Viết câu lệnh BACKUP hoàn chỉnh; Lệnh RESTORE đổi tên database vận dụng thành thạo từ khóa MOVE di dời file chính xác. Lệnh BACKUP chạy tốt, lệnh RESTORE tạo tên mới đúng cấu trúc nhưng quên tùy chọn MOVE gây lỗi ghi đè file cũ đang chạy. Chỉ hoàn thành được lệnh BACKUP dữ liệu thông thường, không nắm được cú pháp lệnh khôi phục đổi thực thể di trú dữ liệu. Viết sai tên thiết bị lưu trữ lưu file (DISK) hoặc sai đường dẫn thư mục vật lý chứa file backup. Không hoàn thành nội dung bài tập quản trị sao lưu phục hồi dữ liệu hệ thống.
Quản lý giao tác an toàn dữ liệu ACID (20%) Sử dụng cấu trúc khối TRY...CATCH hoàn hảo, các lệnh chốt COMMIT hoặc hủy ROLLBACK đặt chuẩn vị trí logic an toàn dữ liệu. Giao tác thực thi điều khiển hủy lệnh đúng bản chất dữ liệu khi có lỗi nhưng bỏ quên không bọc cấu trúc khối TRY...CATCH. Giao tác thiết kế đặt sai vị trí lệnh điều khiển BEGIN TRANSACTION dẫn đến tiến trình hủy bỏ lệnh không toàn vẹn đối tượng phụ thuộc. Thiếu hụt câu lệnh COMMIT hoặc ROLLBACK khiến phiên làm việc bị treo lock độc quyền chiếm dụng tài nguyên hệ thống máy chủ. Không hiểu bản chất giao tác an toàn hoặc viết đoạn script dính lỗi cú pháp liên tục.
Vận dụng giải quyết hiện tượng tranh chấp phiên (10%) Xác định chuẩn xác từ khóa cô lập SERIALIZABLE và triển khai câu lệnh cấu hình thiết lập mức cô lập hệ thống chính xác. Nêu đúng tên mức độ cô lập SERIALIZABLE giúp chống Phantom tốt nhưng cú pháp câu lệnh T-SQL thiết lập bị viết sai từ khóa. Nêu sai tên mức cô lập dữ liệu (Nhầm sang mức độ cô lập REPEATABLE READ hoặc READ COMMITTED chỉ chống được dữ liệu rác). Không hiểu bản chất hiện tượng bóng ma dữ liệu và cách thức hoạt động khóa của hệ quản trị cơ sở dữ liệu. Bỏ trống không hoàn thành câu hỏi lý thuyết vận dụng xử lý tranh chấp phiên chạy đồng thời.
